# Migrations

每次更改数据库结构时，都应该添加迁移脚本，否则程序无法运行。

下面详细介绍 Aphanite 的迁移系统。

## 数据模型的权威来源

由于 Aphanite 并没有使用 Toasty 的迁移系统，数据模型并没有一个“唯一的”权威来源。

但是，我们约定，数据模型的真实类型以 Rust 代码中的类型为主；迁移脚本是为了让现存数据库中的数据模型“长得像” Rust 代码中的类型设计的。

## 迁移文件

Aphanite 的迁移脚本是存放在文件中的 SQL 语句，其中由于我们支持 Postgres 和 SQLite 两种数据库后端，每次迁移都应该为两种方言分别编写迁移脚本。

> [!IMPORTANT]
>
> **永远**不要更改历史上的迁移文件。任何需要修改数据表结构的需求，请添加新迁移文件。

### 文件名

```
[数字 ID]-[迁移名称].sql
```

其中

- 数字 ID：虽然不做限制，但原则上应为四位数。最大值是 65535。**迁移脚本的编写者应该严格保证该数值大于历史上所有迁移脚本的数值**，否则会导致逻辑错误。
- 迁移名称：应为纯 ASCII，以 kebab-case 命名的**唯一**迁移名称。

注意这两项都**不能重复**，否则无法通过编译。

### 存放路径

例如，我添加了名为 `0002-example.sql` 的迁移，那么目前的目录结构里，应该包含：

```
migrations/
├── postgres
│   └── 0002-example.sql
└── sqlite
    └── 0002-example.sql
```

也就是说，每一次迁移，都应该保证 `postgres` 和 `sqlite` 目录下**均有同名文件**，否则编译会失败。

### 文件格式

迁移脚本使用目标数据库对应的 SQL 方言编写。支持以两个连字符 `--` 开头的注释。

但是注意：**不应该在迁移脚本中写 `BEGIN TRANSCATION` 和 `COMMIT TRANSCATION` 语句，Aphanite 的迁移系统会自动加上**。
